Sorting & Searching in Kruskal's

Role of sorting, Union-Find as search, impact analysis with visual comparison, all interactive controls

📋 Why Sorting is Essential

Kruskal's greedy strategy requires edges in sorted order. The algorithm relies on always picking the current global minimum weight edge. If edges are unsorted:

  • We might pick a heavy edge when a lighter valid one exists
  • The greedy choice property breaks — local ≠ global optimal
  • We get a spanning tree, but NOT the minimum one

Sort cost: O(E log E) — the algorithm's dominant step.

🔍 Union-Find as a Search Structure

For each edge (u,v), we search: "Are u and v already connected?" This is the FIND operation.

  • FIND(x): Traverse the forest upward to find x's root (component representative)
  • Path Compression: During FIND, point every visited node directly to the root — flattens the tree
  • Union by Rank: When merging, attach the shorter tree under the taller — keeps depth O(log V)
  • Combined effect: each FIND/UNION ≈ O(1) amortized

🏷️ Paradigm: GREEDY ALGORITHM

Kruskal's is a Greedy Algorithm. At every step it makes the locally optimal choice: pick the lightest edge that does not create a cycle. This local greediness leads to the globally optimal MST, proven by the Cut Property theorem.

✂️ The Cut Property — Why Greedy is Correct

Cut Property (Theorem)

For any cut (S, V–S) of the graph, the minimum-weight edge crossing the cut must appear in some MST.

Why Kruskal's is Correct

When Kruskal's picks edge (u,v): u is in component S, v is in component V–S. This is the lightest crossing edge for that cut. By the Cut Property → it belongs in the MST. ✓

In the diagram:

Blue region = set S. Red region = V–S. Yellow edges = crossing edges. Green edge = minimum crossing — Kruskal's always picks this one.

⚠️ Counterexample — Greedy WITH cycle check vs WITHOUT

✅ Kruskal's (with Union-Find) — Correct MST

Picks lightest edges, skips any that form cycles. Result: a tree.

❌ Naïve Greedy (no cycle check) — WRONG

Adds every light edge blindly. Creates cycles → not a tree!

Kruskal's MST — Full Educational Reference

Algorithm overview, worked example, Union-Find detail, common mistakes

What is a Minimum Spanning Tree?

A spanning tree of G=(V,E) is a connected subgraph with no cycles using exactly V–1 edges. A Minimum Spanning Tree minimizes the total edge weight among all spanning trees.

Applications: minimum-cost network wiring, road planning, cluster analysis, image segmentation.

The Algorithm (Kruskal, 1956)

  1. Sort all edges by weight ascending
  2. Each vertex = its own component (MAKE-SET)
  3. For each edge (u,v,w) lightest first:
  4. If FIND(u) ≠ FIND(v): add to MST, UNION(u,v)
  5. Else: skip (cycle)
  6. Stop when MST has V–1 edges

Worked Example: A–B:4, B–C:2, A–C:6, B–D:5, C–D:3

Union-Find Data Structure

  • MAKE-SET(v): parent[v]=v, rank[v]=0
  • FIND(v): Return root of v's set. Path compression: on every FIND, point all visited nodes directly to root.
  • UNION(u,v): Merge two sets. Union by rank: attach shorter tree under taller. Keeps max depth O(log V).
  • Combined: FIND and UNION both run in amortized O(α(V)) ≈ O(1)

⚠️ Common Mistakes

  • No cycle check → Creates a cyclic graph, not a tree
  • Directed graph → Kruskal's only works on undirected graphs
  • Stopping too early → Need exactly V–1 edges in MST
  • Confused with Prim's → Kruskal = edge-centric; Prim = vertex-centric; both give MST
  • Skipping sort → Produces a spanning tree but NOT the minimum one
